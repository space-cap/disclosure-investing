# 10. 서비스형 UI 개선 계획서

## 1. 현재 문제 인식

현재 Streamlit 화면은 기능 검증용 MVP에 가깝습니다. 공시 수집, 분류, 요약, 운영 상태 확인 기능은 들어왔지만, 화면 완성도는 서비스로 보기 어렵습니다.

문제는 단순히 "예쁘지 않다"가 아닙니다. 사용자가 매일 장 전후에 빠르게 판단해야 하는 도구인데, 정보의 우선순위, 시각적 안정감, 반복 사용성, 신뢰감이 부족합니다.

현재 UI의 핵심 문제는 다음입니다.

- 화면 밀도가 고르지 않다.
- 중요한 공시와 덜 중요한 공시가 한눈에 구분되지 않는다.
- 카드, 표, 상세 영역의 시각 규칙이 약하다.
- 자동 실행 상태와 공시 분석 화면이 서비스처럼 연결되어 보이지 않는다.
- Streamlit 기본 컴포넌트 느낌이 강해서 내부 실험 도구처럼 보인다.
- 사용자가 "오늘 뭘 봐야 하는지" 바로 알기 어렵다.
- 모바일 또는 좁은 화면 사용성을 고려하지 않았다.
- 브랜드/제품 정체성이 없다.

## 2. 목표

UI 개선의 목표는 예쁜 대시보드가 아니라, 매일 반복해서 쓰기 좋은 공시 검토 워크스페이스를 만드는 것입니다.

목표 상태:

- 장 시작 전 1분 안에 오늘 위험 공시를 확인할 수 있다.
- 장 마감 후 그날의 핵심 공시를 빠르게 복기할 수 있다.
- 자동 실행이 정상인지 즉시 확인할 수 있다.
- 공시별 숫자, AI 요약, 위험 요인, 사용자 메모가 자연스럽게 연결된다.
- 위험 공시는 강하게, 관심 후보는 차분하게, 무시할 공시는 뒤로 밀린다.
- 화면이 투자 보조 도구답게 조용하고 신뢰감 있게 보인다.

## 3. 제품 방향

이 도구는 투자 커뮤니티 앱도 아니고, 증권사 HTS도 아닙니다.

지향점:

```text
공시 기반 투자 검토 워크스페이스
```

피해야 할 방향:

- 과한 그래픽
- 마케팅 랜딩 페이지 같은 구성
- 색깔 많은 장식형 대시보드
- 모든 정보를 한 화면에 때려 넣는 HTS식 화면
- AI가 매수/매도 추천하는 것처럼 보이는 UI

가져가야 할 방향:

- 조용한 업무 도구
- 명확한 우선순위
- 숫자 중심
- 위험 우선
- 복기 가능한 기록
- 사람이 최종 판단한다는 구조

## 4. 정보 구조 재설계

현재 탭:

```text
공시 모니터
일일 요약
운영 상태
```

서비스형 UI에서는 다음 구조가 더 좋습니다.

```text
오늘
공시함
관심/메모
리포트
운영
```

### 4.1 오늘

앱의 첫 화면입니다. 사용자가 들어오자마자 봐야 할 내용만 보여줍니다.

구성:

- 오늘 자동 실행 상태
- 오늘 전체 공시 수
- 위험 공시 수
- 단기 이벤트 후보 수
- 수동검토 필요 수
- 오늘 가장 먼저 볼 공시 5개
- 원문/AI 처리 대기 상태

이 화면의 목적:

```text
오늘 뭘 먼저 봐야 하지?
```

### 4.2 공시함

전체 공시 목록과 상세 분석을 보는 작업 화면입니다.

구성:

- 좌측: 필터와 공시 리스트
- 중앙: 공시 상세
- 우측: AI 요약, 확인 포인트, 내 메모

현재 Streamlit 화면은 리스트와 상세가 나뉘어 있지만, 정보의 위계가 약합니다. 공시함에서는 사용자가 공시를 하나씩 검토하는 흐름을 기준으로 재배치해야 합니다.

### 4.3 관심/메모

사용자가 직접 남긴 판단을 모아보는 화면입니다.

구성:

- watch
- reviewed
- avoid
- archived
- 사용자 메모 검색
- 나중에 확인할 공시

이 화면은 공시 검토가 단발로 끝나지 않고 복기로 이어지게 만드는 핵심입니다.

### 4.4 리포트

일일 요약 리포트를 보는 화면입니다.

구성:

- 날짜 선택
- 위험 공시
- 단기 이벤트 후보
- 중장기 관심 후보
- 수동검토 필요
- Markdown 다운로드

현재 `일일 요약` 탭은 이 방향으로 이미 시작했지만, 카드 품질과 레이아웃을 더 다듬어야 합니다.

### 4.5 운영

자동 실행과 오류 상태를 보는 화면입니다.

구성:

- 마지막 실행 성공/실패
- 다음 실행 예정 시간
- 최근 실행 이력
- 원문 수집 실패 목록
- AI 분류 대기/실패 목록
- 로그 파일 위치

운영 화면은 개발자용처럼 보이면 안 됩니다. 사용자가 "오늘 자동 수집이 제대로 됐나?"를 확인하는 서비스 상태판이어야 합니다.

## 5. 첫 화면 설계

첫 화면은 `오늘`이어야 합니다.

### 5.1 상단 상태 바

표시 항목:

```text
오늘 2026-06-16
자동 실행 성공
마지막 실행 16:10
전체 114
위험 6
단기 후보 6
AI 대기 97
```

디자인:

- 한 줄 상태 바
- 성공은 녹색 계열, 실패는 붉은 계열
- 숫자는 작지만 선명하게
- 과한 카드 남발 금지

### 5.2 우선 검토 리스트

가장 중요한 영역입니다.

우선순위:

1. 위험 공시
2. AI가 high risk로 본 공시
3. 단기 이벤트 후보 중 숫자가 큰 공시
4. 수동검토 필요 공시
5. 중장기 관심 후보

각 항목에는 다음만 보여줍니다.

```text
회사명
종목코드
공시명
분류
위험도
핵심 숫자
AI 한 줄 요약
상태 변경 버튼
```

### 5.3 오늘 처리 상태

작게 표시합니다.

```text
원문 수집 20건
지표 추출 20건
AI 분류 10건
리포트 생성 완료
```

이 영역은 사용자가 데이터 신뢰도를 판단하는 데 중요합니다.

## 6. 공시 상세 화면 개선

현재 상세 화면은 정보가 세로로 나열되어 있습니다. 서비스형 화면에서는 판단 흐름에 맞게 묶어야 합니다.

권장 구성:

```text
회사/공시 헤더
핵심 판단
핵심 숫자
AI 요약
확인 포인트
원문/정정/오류 정보
내 판단과 메모
```

### 6.1 회사/공시 헤더

포함:

- 회사명
- 종목코드
- 접수일
- 공시명
- DART 원문 링크
- 정정 여부

### 6.2 핵심 판단

가장 먼저 보여야 합니다.

```text
위험 공시
avoid
high
유상증자로 기존 주주 희석 가능성이 큼
```

AI 결과가 없으면 규칙 기반 결과를 표시하고, AI 대기 상태를 명확히 보여줍니다.

### 6.3 핵심 숫자

숫자는 카드보다 "지표 행" 형태가 좋습니다.

예:

```text
계약금액        12.1억
최근매출        235.3억
매출대비        5.13%
계약기간        2026-06-16 ~ 2028-06-30
계약상대방      케이비밸브상사
```

유상증자라면:

```text
신주수          6,000만주
희석률          302.21%
발행가          500원
납입일          2026-07-01
상장예정일      2026-07-15
```

숫자가 없으면 `nan`이나 빈 카드가 아니라, 해당 섹션 자체를 줄이거나 "추출된 핵심 숫자 없음"으로 표시합니다.

## 7. 시각 디자인 원칙

### 7.1 색상

과한 단색 테마를 피하고, 정보 의미에만 색을 씁니다.

권장:

```text
배경: #F6F7F9
표면: #FFFFFF
본문: #172033
보조 텍스트: #667085
라인: #E2E6EE
위험: #B42318
관심: #175CD3
긍정/정상: #067647
주의/수동검토: #93370D
```

색은 장식이 아니라 상태 전달에만 씁니다.

### 7.2 타이포그래피

권장:

- 기본 14px 또는 15px
- 제목 20px 이하
- 카드 내부 제목 16px 이하
- 숫자는 굵게
- 긴 공시명은 2줄까지 허용

피해야 할 것:

- 큰 히어로 제목
- 과한 굵기
- 좁은 카드 안의 큰 글씨
- 버튼 안에서 줄 깨지는 텍스트

### 7.3 카드 사용

카드는 반복 항목에만 씁니다.

좋은 카드:

- 우선 검토 공시
- 최근 실행 이력 요약
- 리포트 섹션 요약

나쁜 카드:

- 페이지 전체를 감싸는 큰 카드
- 카드 안의 카드
- 장식용 카드

### 7.4 표 사용

공시 목록은 표가 맞습니다. 다만 현재보다 열을 줄여야 합니다.

기본 열:

```text
접수일
회사
공시명
분류
위험도
액션
상태
```

DART URL, 내부 ID, 긴 AI 텍스트는 상세 화면으로 보내는 것이 좋습니다.

## 8. Streamlit 유지 여부

### 8.1 Streamlit으로 계속 갈 수 있는 경우

개인용 또는 내부용이라면 Streamlit을 계속 써도 됩니다.

조건:

- CSS를 정리한다.
- 탭 구조를 재설계한다.
- 표와 상세 영역을 더 깔끔하게 만든다.
- 운영 상태 탭을 다듬는다.
- 로그인/권한/다중 사용자 요구가 없다.

Streamlit 개선만으로도 지금보다 훨씬 좋아질 수 있습니다.

### 8.2 서비스로 확장하려면

외부 사용자에게 제공하거나, 장기적으로 진짜 서비스처럼 만들려면 구조를 바꾸는 것이 좋습니다.

권장 구조:

```text
FastAPI
PostgreSQL
React 또는 Next.js
작업 스케줄러
Background worker
```

이 경우 Streamlit은 내부 관리자 도구로 남기고, 사용자용 화면은 별도 웹앱으로 만드는 것이 좋습니다.

## 9. 단계별 개선 계획

### 1단계: Streamlit UI 정리

목표:

현재 기능을 유지하면서 화면을 서비스형 내부 도구 수준으로 올립니다.

작업:

- `오늘` 탭 추가
- 기존 `공시 모니터`를 `공시함`으로 이름 변경
- `일일 요약`을 `리포트`로 이름 변경
- `운영 상태`를 `운영`으로 이름 변경
- 상단 KPI 영역 재설계
- 우선 검토 리스트 추가
- 상세 화면 섹션 정리
- 표 컬럼 축소
- CSS 파일 분리 검토

예상 효과:

- 바로 쓰기 좋아진다.
- 개발 비용이 낮다.
- 현재 구조를 크게 흔들지 않는다.

### 2단계: 사용자 흐름 개선

목표:

공시를 보고 판단을 남기는 흐름을 자연스럽게 만듭니다.

작업:

- `watch`, `reviewed`, `avoid`, `archived` 상태 버튼화
- 메모 저장 UX 개선
- 관심/메모 전용 화면 추가
- 오늘의 우선 검토 공시 자동 선정
- 위험 공시 먼저 보기 기본값 적용

예상 효과:

- 매일 반복 사용성이 좋아진다.
- 투자 복기 데이터가 쌓인다.

### 3단계: 데이터 신뢰도 표시

목표:

사용자가 분석 결과의 완성도를 판단할 수 있게 합니다.

작업:

- 원문 수집 상태 표시 개선
- AI 분류 대기/실패 표시
- 숫자 추출 성공 여부 표시
- 정정공시 강조
- 자동 실행 이력과 화면 데이터 연결

예상 효과:

- 사용자가 결과를 맹신하지 않고 검토할 수 있다.
- 자동화 도구에 대한 신뢰가 올라간다.

### 4단계: React/Next.js 전환 검토

목표:

서비스형 UI로 확장할지 판단합니다.

전환 기준:

- 외부 사용자에게 보여줘야 한다.
- 모바일 대응이 필요하다.
- 로그인/권한이 필요하다.
- 대시보드 상호작용이 많아진다.
- Streamlit CSS 커스터마이징이 한계에 온다.

전환한다면:

```text
API: FastAPI
DB: PostgreSQL
Frontend: Next.js
Worker: scheduled Python job
```

## 10. 권장 다음 개발 작업

가장 먼저 할 일은 React 전환이 아닙니다. 지금은 제품 흐름이 아직 완전히 굳지 않았기 때문에, Streamlit에서 화면 구조를 먼저 다듬는 것이 좋습니다.

추천 순서:

1. `오늘` 탭 추가
2. 탭 이름 정리
3. 상단 KPI와 자동 실행 상태 통합
4. 우선 검토 공시 리스트 추가
5. 공시 상세 화면 섹션 재배치
6. 표 컬럼 축소
7. 관심/메모 화면 추가
8. 이후 React/Next.js 전환 여부 결정

## 11. 성공 기준

UI 개선 후 다음 기준을 만족해야 합니다.

- 첫 화면에서 오늘 위험 공시 수와 우선 검토 공시를 바로 알 수 있다.
- 공시 하나를 클릭하면 핵심 숫자와 AI 요약이 5초 안에 읽힌다.
- 사용자가 공시에 대해 `watch`, `avoid`, `reviewed` 판단을 쉽게 남길 수 있다.
- 자동 실행 성공 여부가 별도 설명 없이 보인다.
- 숫자가 없는 공시는 어색한 빈 카드나 `nan` 없이 표시된다.
- 화면이 내부 실험 도구가 아니라 매일 쓰는 업무 도구처럼 보인다.

## 12. 결론

현재 UI가 허접해 보이는 이유는 기능이 부족해서가 아니라, 제품 흐름과 정보 위계가 아직 정리되지 않았기 때문입니다.

지금 당장 해야 할 일은 색을 더 칠하는 것이 아니라, 사용자의 하루 루틴에 맞게 화면을 다시 짜는 것입니다.

우선은 Streamlit 안에서 다음 형태로 재구성합니다.

```text
오늘 -> 공시함 -> 관심/메모 -> 리포트 -> 운영
```

이 구조가 안정되면, 그때 React/Next.js 기반 서비스형 웹앱으로 전환하는 것이 좋습니다.
